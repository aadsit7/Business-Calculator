Application Workspace partner economics

Application work is counted the way it is actually performed today — once per application, per customer, per version. Application Workspace collapses that to distinct packages built once and released to every zone, with the Setup Store supplying prepared sources for managed applications.

Why this model is built this way

The structural correction versus a flat effort-reduction model.

What a flat model gets wrong

  • It counts application work as one queue, so the same application rebuilt in ten tenants looks like one package rather than ten.
  • It applies a percentage reduction to packaging effort, which cannot express the actual mechanism: the connector supplies a prepared source and the partner validates one version for the whole practice.
  • It hides the update treadmill, where most repeated cost lives — build cost is paid once per app, update cost is paid every version, every tenant.
  • It prices license across the full user base while measuring savings against a small flat workload, which understates the case at exactly the point a partner is deciding.

What this model does instead

  • Counts today's work as application x tenant x version, which is how it is actually performed.
  • Splits third-party from custom, because only third-party apps have a managed-package path and cross-customer reuse.
  • Pays validation once per version across all zones, and charges a separate per-zone release and entitlement cost so "click and deploy" is never modeled as free.
  • Keeps testing, approval, rollout policy, exceptions, and skilled custom packaging as permanent partner-owned work.

Product facts the model respects

Taken from the partner guide and Recast documentation.

Positioning and scope

  • Application Workspace is the application control and access layer across existing endpoint, workspace, identity, and delivery platforms. It does not replace Intune, Configuration Manager, Citrix, AVD, Windows 365, or identity.
  • The Setup Store is a connector-based administrative source of prepared application resources. It is not the customer's Catalog.
  • A managed package is source-connected and can receive new versions through the connector. The partner still owns testing, approval, and rollout policy.
  • Custom and unmanaged packages remain partner-controlled for internal, legacy, unsupported, or heavily customized applications.
  • Catalog is the user-facing storefront for entitled apps. Workspace is the user's active shelf. Publishing modes are Hidden, Catalog, Workspace, and Forced.
  • Multiple zones support separate customers; each zone requires licensing.

Licensing and commercial truth

  • Release & Patch Management is a separate add-on and is required for full Setup Store connector capability and release-stage functions. The base license carries a limited selection.
  • Access Manager is licensed separately.
  • Do not assume every connector, Setup Store workflow, access feature, patch workflow, or release-stage function is included. Confirm edition and current licensing.
  • Margin recovery must be measured against a baseline. Managed-service inclusions depend on licensed capabilities.
  • Project acceleration applies only where application readiness is on the critical path. It cannot remove unrelated security, infrastructure, licensing, or business delays.
  • Capacity is modeled as reallocation or avoided hiring, not as headcount removal.
