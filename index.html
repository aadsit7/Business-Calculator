Model the operating system, not just the software.

1Run a stratified bake-off

Gate 1 Approve the baseline
Week 1

Pull volumes, touch time, rework, tickets and elapsed time from the queue, tooling and timesheets.

Exit: delivery owner signs off on the numbers to beat.
Gate 2 Select representative applications
Week 1

Use covered standard apps, frequently updated apps, custom/complex apps, multi-target apps and known problem apps.

Exit: sponsor confirms the sample represents the operating book.
Gate 3 Run both workflows
Weeks 2–3

Measure engineer touch time, elapsed time, rework, success, exception effort and addressable support demand.

Exit: both teams accept the measurement method.
Gate 4 Replace assumptions
Week 4

Enter measured current and future values below. The decision view uses measured results when supplied.

Exit: approve, revise or stop based on evidence.

Method, boundaries and official product references

Model boundaries. This is a planning and proof tool, not a promise of savings. It does not treat all software tickets as addressable, all returned capacity as cash savings, or customer schedule value as MSP profit. New-service contribution margin is kept separate from existing-contract operating value.

Official product references used to constrain capability language:

  • Application Workspace End User Guide — Workspace, Catalog and user interaction.
  • Setup Store Connector — managed and unmanaged packages; licensing caveat.
  • Create a managed package — managed packages receive later product versions through the connector.
  • Deploy using Microsoft Intune — documented deployment workflow.
  • Application Workspace 4.4 release — reporting data categories and current release context.
  • Application Workspace components — zones, server, satellite, agent and architecture boundaries.
